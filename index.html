The Repository

A git repository is a folder on your computer

Your code and all its versions live in the repository

Two ways to get a repository

  • make a new one
  • copy an old one

git init makes a new repository

The .git folder holds git's gits guts

See it with ls -A

Don't mess with the .git folder,
it stores all of your changes

The current folder becomes the repository

It's easy to accidentally turn the wrong folder into a repository!

Step-by-step

  1. mkdir - make a new folder
  2. cd - enter the folder
  3. git init - make the repo

git clone [path] copies a repository

This makes a new folder

This is different than git init!

You can specify which folder with
git clone [path] [destination]

The [path] can be:

  • a folder on your computer
  • a link to a repository

Step-by-step

  1. cd into parent folder
  2. git clone [path]

https://github.com/Lionex/github-workshop.git

git status tells you about your repository

git status can suggest what to do next

Staging & Committing

Staging combines related changes

Commits document those changes
file-by-file, line-by-line

A commit stores

  • a descriptive comment
  • metadata
  • changed lines
  • commit id

git add [file] stages changes

git add -p [file] stages parts of a file

[file] can be:

  • a file
  • a list of files
  • a folder

git commit puts a change in the history

Step-by-step

  1. create or change a file
  2. git status - see changes
  3. git add [file] - stage
  4. git commit - write history
git has other commands to make the process easier

git reset HEAD -- unstages changes

git checkout [file] removes unstaged changes

Useful when you want to return to a clean version after making bad changes

git checkout . removes all changes

git diff shows unstaged changes
file-by-file, line-by-line

Pro tips for Pro commit messages

Write in present tense, imperative mood

  • Add README with title and summary
  • Refactor presentation section on commit messages

Commits should sound like commands

Describe not just what, but why.

One line usually isn't enough

  • Short summary line
  • Detailed body

Almost like an email

Fix cost propagation bug

Previously, cost was calculated but not passed to child
nodes resulting in failure to create optimal paths.

Ensure child nodes receive calculated cost to prevent
sub-optimality and failure to converge in results.

Limit your summary to about 50 characters

~70 characters for the body

git has a rich set of tools
for working with history

Lets look at logging & reverting

git log shows history

The log prints to terminal the:

  • unique commit id
  • metadata such as author
  • commit message

The log doesn't show

  • which files changed
  • what lines changed
  • the history of one file

unless we ask it to

git log --stat adds files changed

git log --patch adds lines changed

git log --follow [file]
shows [file]'s history

Sometimes we need less information

git log --oneline one line per commit

git log -[n] shows last [n] commits

git revert undoes commits

A revert makes a new commit,
reversing an old one

Step-by-step

  1. make a commit
  2. git log -1 - show last commit
  3. copy the commit id
  4. git revert [id] - revert
  5. git log --patch -2 - inspect

Branches

In git, history isn't a straight line,
but a branching story

Chronological order isn't enough for something as complicated as code

In practice, we work on multiple things at once

Seperate unrelated changes

  • one feature, one branch
  • one bug fix, one branch
  • one release, one branch

Good commits are only as good
as your branches!!!

git branch lists branches

The default branch is master

git branch [name] makes branch [name]

The branch splits from the current commit

When you create a branch,
you do not move to it.

git checkout [ref] travels through history

We are still in the old branch

We need to move to the new branch explicitly

[ref] can be:

  • branch name
  • commit [id]

A checkout will change the files in your folder

Your files are the same as they where in that point in history

You need to have a clean repo, aka no staged or unstaged changes

Step-by-step

  1. git branch [name]
  2. git checkout [name]
  3. make a commit
  1. make a new branch from the current commit
  2. checkout the new branch we made
  3. add a commit to the new branch

Merging

We need a way to join & integrate changes

git merge [target] merges [target]
into the current branch

Step-by-Step

  1. git branch - list branches
  2. git checkout [base]
  3. git merge [target]
  4. git log --oneline

A merge can either:

  • make a new commit
  • fast-forward

merge commit

fast-forward

Sometimes we have a merge conflict

Both branches changed the same lines

# README: This is a practice `git` repository

<<<<<<< HEAD
I added this change in master, no conflicts here!
=======
I added this change in conflict, this is a conflict!
>>>>>>> conflict

git helps visualize conflicts

git does not presume which changes are right

git add the fixed file

# README: This is a practice `git` repository

I added this change in master, but there was a conflict!

Step-by-step

  1. make a commit in a new [branch]
  2. commit changing same line in [base]
  3. git merge [branch]
  4. Fix the conflict in [file]
  5. git add [file] && git commit

git reset --hard HEAD cancels a merge with conflicts

Rebase

Transplant branches from one place to another

Avoid new commits!

git rebase [ref] transplants onto [ref]

Rewind changes and re-apply one-by-one

Fix conflicts commit-by-commit

git rebase --abort stop a rebase in progress

git rebase -Xtheirs automatically favor
rebase-branch changes

git rebase -Xours automatically favor
base branch changes

Note: this is the opposite behavior compared to git merge -X
